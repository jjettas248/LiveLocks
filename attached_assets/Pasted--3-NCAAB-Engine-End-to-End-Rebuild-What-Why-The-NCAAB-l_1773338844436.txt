#3 - NCAAB Engine End-to-End Rebuild

What & Why
The NCAAB live engine has three related problems: (1) projection math sometimes diverges from what gets displayed, (2) the pick direction (Over/Under) can contradict the actual projection, and (3) probabilities lack calibration, producing inflated 80–90% edges that don't reflect reality. The goal is to rebuild the engine in a modular, auditable way so that every displayed value traces directly back to the underlying calculation, with no silent contradictions.

Important implementation rule
Treat this task as a server-side engine integrity rebuild first.
Do not redesign UI, add non-essential abstractions, or introduce client-side recomputation until:
1. projection math is correct
2. probabilities are calibrated
3. pick direction is consistent
4. output validation tests pass

Done looks like
- The displayed projection, probability, and pick direction are always mathematically consistent with each other
- An Over is never recommended when the projection is clearly below the line, and vice versa
- Probabilities rarely exceed realistic thresholds for NCAAB totals/spreads, with far fewer fake 80–90% edges
- Full game total, 1H total, 2H total, spread, team total, 1H team total, and 2H team total all use appropriate game-level accumulation logic
- Every settled play logs the full diagnostic chain: line, projection, actual result, displayed side, actual result, raw probability, calibrated probability
- A validation function catches contradictions at output time and surfaces them as console warnings
- The client renders only from the final typed engine output returned by the server, with no client-side recomputation of projection, edge, probability, or side
- Test cases covering the 5 defined contradiction scenarios all pass

Out of scope
- Changing the data sources (ESPN, SGO, Odds API, Torvik) — those are fine
- NCAAB player prop lines — this task is for the game-level total/spread/team-total engine only
- Probability calibration foundations already covered in Task #1 — this task extends that work into projection, display, and pick-direction consistency
- Redesigning the NBA tab

4. Rebuild the projection layer as a pure function
Extract pace calculation, half blending, team-total split, expected possessions remaining, and margin projection into a single:

calculateNCAABProjection(input: NCAABGameInput)

This function must return named intermediate values:
- baseProjection
- paceAdjustment
- halfBlend
- possessionAdjustment
- marginAdjustment
- finalProjectedTotal
- finalProjectedSpread
- finalProjectedTeamTotalA
- finalProjectedTeamTotalB

Rules:
- all modifiers must be bounded
- no free stacking
- no hidden mutation
- projections must be debuggable and reproducible from the returned intermediate values

5. Rebuild the probability layer
Extract probability generation into:

calculateNCAABProbabilities(input, projectionResult)

Replace the current linear formula like:
50 + diff * mult * 0.3

Use a realistic distribution-based approximation for NCAAB totals/spreads, with a variance estimate appropriate for college basketball scoring volatility.

Return:
- rawOverProb
- rawUnderProb
- rawSpreadProb
- raw1HProb
- raw2HProb

Then apply:
calibrateNCAABProbability(rawProbability, marketType, context)

Rules:
- dynamic multiplier must be applied consistently
- sanitizeProb() should only clip final values, not hide broken math
- calibrated probabilities for totals/spreads should rarely exceed 75–80% unless historical calibration supports it
- any probability above threshold should emit a warning into debug payload

6. Fix pick-direction logic
Create:

determineRecommendedSide(projectedValue, line, overProb, underProb, edgeThreshold)

This function must:
- explicitly compare calibrated over vs under probability
- explicitly compare projection vs line
- reject contradictions
- return only: "OVER" | "UNDER" | "NO_EDGE"

Examples of contradictions to catch:
- projection below line but side = OVER
- projection above line but side = UNDER
- tiny edge but side not NO_EDGE
- over probability shown when under side is recommended

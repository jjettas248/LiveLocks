#44 - Enforce Engine Probability as Single Source of Truth
What & Why
Guarantee that every probability displayed anywhere in the UI comes directly from the engine model output, and that TypeScript structurally prevents sportsbook-implied probabilities from ever reaching a display component — even if future code changes occur.

Do NOT modify:

calibrateProbability()
canonical market logic
API route signatures
NCAAB or MLB engines
Done looks like
calculateProbability() in server/storage.ts returns impliedProbability: probability (engine value) on all return paths; marketImpliedProb is used only internally for edge math
impliedProbability is removed from all client-facing TypeScript types in shared/schema.ts so the compiler will error if any UI component attempts to access it
ProbabilityRing component accepts only a probability: number prop — no impliedProbability or marketProbability props exist
Dashboard probability rings, edge cards, halftime play cards, and quick signal badges all display play.probability
Matchup calculator output displays result.probability
MLB live page probability displays use play.probability
Parlay slip probability displays use leg.probability
A dev-only null assertion guard is added in the dashboard play card render loop (if (play.probability == null) console.error(...))
No temporary console.log debug lines remain in production code
TypeScript compiles with no errors
Out of scope
Changes to calibrateProbability() or its logic
Changes to canonical market logic
Changes to API route signatures
NCAAB or MLB engine logic
Tasks
Lock API return structure — In server/storage.ts calculateProbability(), confirm all return paths set impliedProbability: probability (the engine value). marketImpliedProb must only appear in the edge calculation (edge = probability - marketImpliedProb), never returned separately for UI consumption.

Remove impliedProbability from client-facing types — In shared/schema.ts, remove impliedProbability from any frontend-facing play/result interface or type. The backend may still compute it internally, but the TypeScript type used by UI components must not expose it. This makes TypeScript enforce the constraint structurally.

Audit & remove sportsbook field usage from frontend — Search all frontend files for marketImpliedProbability, bookProbability, sportsbookProbability, impliedProbability. If any appear in JSX rendering, replace with play.probability or result.probability. These fields may remain in non-display edge math logic only.

Restrict ProbabilityRing props — In client/src/components/probability-ring.tsx, update the props type to only accept probability: number. Remove any impliedProbability or marketProbability props if they exist. Confirm all call sites pass play.probability.

Enforce engine probability in dashboard display — In client/src/pages/dashboard.tsx, confirm probability rings, edge cards, halftime play cards, and quick signal badges all use play.probability. Fix any that reference implied or sportsbook fields. Replace any result.impliedProbability display in the matchup calculator with result.probability. Add the dev-only guard: if (process.env.NODE_ENV !== "production" && play.probability == null) { console.error("Missing engine probability for play", play) }.

Validate MLB live page and parlay slip — In client/src/pages/mlb-live.tsx, confirm all probability display elements use play.probability. In client/src/components/parlay-slip.tsx, confirm the probability display uses leg.probability.

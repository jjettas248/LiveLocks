#66 - Free Tier: Daily Play Reset & Conversion UX

## SYSTEM
Conversion System / UX Layer / Access Control

## OBJECTIVE
Replace the current lifetime 3-play free cap with a daily 3-play limit that resets automatically by calendar date. Add conversion UX that makes the free tier valuable enough to validate the product while preserving clear upgrade pressure.

This task must:
- create a daily activation loop
- preserve unlimited access for paid tiers
- add visible locked-value UX for free users
- avoid any Stripe, pricing, or alerting changes

---

## CONTEXT
Current issue:
- Free users are capped at 3 lifetime plays
- This creates a dead-end instead of a daily habit loop
- Many users never activate enough to understand product value
- Checkout / Stripe logic is being handled separately and must not be touched here

This task replaces lifetime-gated free access with daily-gated free access.

---

## PRODUCT DECISION
Use:
- 3 free plays per day
- automatic daily reset by calendar date
- no account-level expiration
- no 5-day trial timer
- no missed-play alerts
- no SMS/email automation changes in this task

Paid users:
- Pro / All Sports remain unlimited and unchanged

---

## REQUIREMENTS

### 1) Schema migration
Add the following columns to the users table:
- `playsUsedToday` integer NOT NULL default 0
- `playsResetDate` text nullable or non-null if required by current patterns, storing an ISO calendar date string like `2026-03-19`

Requirements:
- write the Drizzle migration
- update `shared/schema.ts`
- update insert/select typing as needed
- legacy `playsUsed` may remain in the DB for historical/admin purposes, but it must no longer control access

Backfill / initialization rule:
- existing users must safely initialize into the new system
- default daily state should resolve safely to 0 used if no valid reset date exists yet

---

### 2) Daily reset access logic
Refactor free-tier gating in `server/auth.ts` and related storage methods.

Requirements:
- the only source of truth for free-tier access control must be:
  - `playsUsedToday`
  - `playsResetDate`
- on each protected request, compare `playsResetDate` to today’s server-local calendar date string
- if `playsResetDate !== today`, automatically:
  - reset `playsUsedToday` to 0
  - set `playsResetDate = today`
- then enforce the free-tier limit
- increment `playsUsedToday` only when a gated play is actually consumed
- do NOT use legacy `playsUsed` for access control anywhere

Important:
- use one consistent server-local date strategy everywhere
- do not introduce UTC/server mismatch logic unless already standardized in the app
- keep `sentWall` behavior unchanged so the existing wall email still fires at most once per user

---

### 3) Storage layer updates
Update `server/storage.ts` so all access-related methods use the new daily model.

Requirements:
- refactor any increment method to update `playsUsedToday`
- add/update helper methods as needed to support daily reset safely
- prevent mixed usage of legacy `playsUsed` and new daily fields for gating
- retain compatibility with existing admin reset tooling if it already exists, but do not build new admin tooling in this task

---

### 4) Auth response contract

Requirements:
- include:
  - `playsUsedToday`
  - `playsResetDate`
- frontend auth state must read daily usage from these fields
- if legacy `playsUsed` must remain in the response temporarily for compatibility, it must not drive gate logic or primary dashboard display

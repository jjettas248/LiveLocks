# TASK: #84 - Persisted Plays Grading + Calibration Dashboard (Final Hardened Spec)

## SYSTEM:
Analytics + Retention (strictly isolated from Model Performance)

## OBJECTIVE:
Grade all persisted plays after their games go FINAL and expose the results in a standalone calibration dashboard. This is a pure analytics layer and MUST remain completely isolated from model performance tables, KPIs, and optimization loops.

## WHAT & WHY:
Persisted plays are currently being stored but not resolved. Operators need to know whether these called plays actually hit, and they need a separate calibration view to evaluate real-world edge and probability behavior without contaminating official model performance metrics.

## DONE LOOKS LIKE:
- Every persisted play tied to a FINAL game has result, finalStat, and settledAt filled in automatically
- A new admin-only GET /api/persisted-plays/calibration endpoint returns graded plays with filters and aggregated metrics
- A new admin-only Calibration tab shows:
  - sortable table of graded plays
  - summary metrics
  - bucket breakdowns
  - calibration curve chart
- Persisted plays grading is fully isolated from all model performance tables and KPIs
- Cron-driven grading is idempotent and safe to run repeatedly
- Unsupported or unresolved plays are logged clearly and skipped without crashing the server

## OUT OF SCOPE:
- Any engine logic changes
- Any model training / optimization changes
- Any changes to how plays are originally recorded
- Any changes to alert timing
- Any grading of halftime_play_alerts
- Any modifications to existing /api/plays/* endpoints

## REQUIRED IMPLEMENTATION:

### 1. SCHEMA HARDENING
In shared/schema.ts, ensure persisted plays schema includes the following grading fields if not already present:
- result: "WIN" | "LOSS" | "PUSH" | null
- finalStat: numeric | null
- settledAt: timestamp | null

If these fields do not exist, add them and create the migration.

Also confirm persisted plays store the canonical identifiers required for deterministic grading:
- playerId (DB player ID)
- sport
- market
- direction
- line
- gameId OR externalGameId used to fetch the correct final game

Do NOT rely on player name + date inference if a game identifier can be stored and used directly.

### 2. GRADING SERVICE
Create:
server/services/gradePersistedPlays.ts

This service must:
- query persisted plays where result IS NULL only
- use playerId only for player matching
- fetch final box scores for the correct game
- only grade when game status === "FINAL"
- compute finalStat for supported markets
- compute WIN / LOSS / PUSH
- persist result, finalStat, and settledAt through existing storage/settle method
- never overwrite already-settled plays

This service must be idempotent.

### 3. SUPPORTED MARKET MAPPING
Support exactly these market types:
- PTS → points
- REB → rebounds
- AST → assists
- P+R → points + rebounds
- P+A → points + assists
- R+A → rebounds + assists
- P+R+A → points + rebounds + assists
- 3PM → made threes

If market is unsupported:
- log it clearly
- skip grading for that play
- do not crash
- do not set a fake result

### 4. RESULT LOGIC
Apply strict settle logic:

For OVER:
- actual > line → WIN
- actual < line → LOSS
- actual == line → PUSH

For UNDER:
- actual < line → WIN
- actual > line → LOSS
- actual == line → PUSH

### 5. PLAYER MATCHING
Use ONLY:
- persisted_play.playerId

Do NOT use:
- player name
- fuzzy matching
- normalized string matching

If playerId is null or unresolved in final stats:
- log the failure
- skip grading for that row
- leave result null

### 6. FINAL GAME FETCHING
Use ESPN final game box score data for the specific game tied to the persisted play.

Requirement:
- grading must resolve against the correct game, not a best guess
- if the play’s game identifier is missing, log and skip
- if final box score is unavailable, skip and retry on next cron run

### 7. SUPERSEDE OLD GRADING PATH
Replace or supersede autoSettlePersistedPlays in server/analyticsResolver.ts with the new service.

There must be one canonical grading path for persisted plays.

### 8. GRADING TRIGGER
Wire gradePersistedPlays() into a cron-style interval in server/index.ts that runs every 3 minutes.

Requirements:
- catch and log errors
- do not crash the server
- safe for repeated execution
- optional additional trigger on game status transition to FINAL is allowed if already observable

### 9. STORAGE WRITE RULES
Use existing settlePlay storage method if available, but enforce:
- only update plays where result is currently null
- populate:
  - result
  - finalStat
  - settledAt
- do not touch model performance tables
- do not backfill into optimization systems

### 10. CALIBRATION API
Add:
GET /api/persisted-plays/calibration

Requirements:
- admin-only
- optional filters:
  - sport
  - market
  - startDate
  - endDate
- return graded plays only
- return summary metrics:
  - totalPlays
  - winRate
  - pushRate
  - avgEdge
  - avgProbability
- return grouped performance by edge bucket:
  - 0–5%
  - 5–10%
  - 10%+
- return grouped performance by probability bucket:
  - 50–60
  - 60–70
  - 70–80
  - 80–100
- MUST NOT read from or join against any model performance table

### 11. BUCKET DEFINITIONS
Define bucket math explicitly:

Edge buckets:
- 0 <= edge < 5
- 5 <= edge < 10
- edge >= 10

Probability buckets:
- 50 <= prob < 60
- 60 <= prob < 70
- 70 <= prob < 80
- 80 <= prob <= 100

Use consistent bucket assignment everywhere: API, UI summaries, and chart data.

### 12. CALIBRATION DASHBOARD UI
Create:
client/src/components/calibration-dashboard.tsx

Add it as an admin-only tab in client/src/pages/admin.tsx

Component must include:

- sortable table with:
  - Date

  - Dir

  - Actual
  - Result

  - WIN = green

- summary panel with:
  - total plays
  - win rate
  - push rate
  - avg edge
  - avg probability
- bucket breakdown panels/tables
- calibration curve chart using Recharts

### 13. CALIBRATION CURVE
Render a calibration chart where:
- X axis = predicted probability bucket midpoint
- Y axis = actual win rate for that bucket

Include:
- reference diagonal line (ideal calibration)
- actual data series from bucketed results

This chart is for operator evaluation only and MUST NOT feed back into the model automatically.

### 14. FAILURE HANDLING
For each unresolved case, log clearly and skip safely:
- missing game identifier
- game not FINAL
- missing final boxscore
- unsupported market
- missing playerId
- player not found in final stats

Do not crash the cron.
Do not write partial/fake results.

### 15. HARD ISOLATION FROM MODEL PERFORMANCE
This is a strict rule:

Persisted plays grading and calibration analytics MUST NOT:
- write to model performance tables
- affect accuracy KPIs
- affect optimization loops
- alter any existing model evaluation endpoints

This is a separate analytics surface only.

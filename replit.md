# LiveLocks by PropPulse — NBA/NCAAB Live Lines

## Overview
LiveLocks is a full-stack Progressive Web Application (PWA) designed for betting analytics in NBA and NCAAB. It provides real-time probabilities for player prop bets, analyzes NCAAB live spread/total/team-total probabilities, and enables users to construct correlation-adjusted parlays. The platform integrates live game data from various sports data providers and offers Stripe-based subscriptions, SMS/push notifications, and an administrative panel. The project's vision is to enhance sports bettors' decision-making with data-driven insights and sophisticated analytical tools.

## System Architecture

### Frontend
- **Frameworks**: React with Vite, Tailwind CSS, and `shadcn/ui`.
- **Data Fetching**: TanStack Query.
- **Forms**: `react-hook-form`.
- **Routing**: `wouter`.
- **UI/UX**: Features a top tab bar for sports (NBA Live, NCAAB Live, MLB Live) with sub-tabs for specific analyses. Subscription tiers control access to certain features (e.g., NCAAB tab for Pro/All Sports users). Includes book filtering, probability gauges, EV verdicts, CLV indicators, and H2H matchup history. A daily slate reset system with an admin-configurable reset time. Mobile-optimized components like a Parlay Slip bottom sheet and pull-to-refresh functionality. PWA updates are handled by a service worker with client-driven `skipWaiting` and a network-first strategy for navigation.
- **Probabilities**: Engine-calibrated probabilities are displayed directly from the server.

### Backend
- **Framework**: Express.js with TypeScript and `tsx`.
- **Database**: PostgreSQL with Drizzle ORM.
- **Shared Components**: `shared/schema.ts` and `shared/routes.ts` for type safety.
- **Authentication**: JWT-based authentication with role-based access control tied to subscription tiers (Free, Pro, All Sports).
- **Notifications**: Twilio for SMS alerts. Resend for transactional emails (welcome, lifecycle, subscription).
- **NBA Probability Engine (v2)**: Distribution-based probability using normal CDF instead of bucket interpolation. Key components in `server/nba/`: `probabilityEngine.ts` (CDF probability, variance model, fragility), `archetypes.ts` (7-level classification: stable_star, stable_starter, volatile_starter, bench_microwave, low_minute_big, lineup_impacted, role_uncertain), `marketFamily.ts` (family grouping, derivative suppression), `directionalBias.ts` (rolling over/under ratio tracking). No post-calibration expansion. Split calibration by single/combo stats, volatile/stable archetypes. Safety ceilings per archetype+stat-type. Family suppression applied at batch level in both live-signals and halftime-plays paths. Parallel odds cache pre-warming. Full diagnostics persistence in persisted_plays (archetype, fragilityScore, familyId, siblingCount/Rank, flagshipOrDerivative, familyPenaltyFactor, calibrationTrack, confidenceCeilingApplied, ceilingReason, rawProbOver/Under, modelEdge, minutesExpected/Variance). Structured ENGINE_OUTPUT, DIRECTIONAL_SPLIT, MARKET_FAMILY logging. Confidence buckets (60-64/65-69/70-74/75-79/80+) with filters for direction, single/combo, archetype, flagship/derivative. Analytics calibration views (row-level, player-game, market-family) at `/api/analytics/calibration-views`.
- **NCAAB Engine**: A single source of truth for NCAAB projections, probabilities, and pick directions, utilizing a normal CDF probability model and deterministic edge rules. Includes a diagnostics module for logging and analysis. Market surfacing uses `qualifiedEdge`/`fallback` architecture: markets with `bookLine != null` are always `available = true`, with `qualifiedEdge` (edgeFrom50 >= 12) controlling strength classification. Top Plays use weighted scoring: `+10` for qualifiedEdge, `+5` for 2H markets, `+5` additional for 2H at halftime, `-3` for isDerived, `-2` for fallback. Per-game guardrail: max 3 markets, max 1 fallback. Color-coded confidence tiers: ELITE (green, edgeFrom50 >= 20), STRONG (teal, >= 12), LEAN (gray, < 12). Halftime cards show yellow "LIVE 2H" badge with 2H line context. Server logs `[NCAAB_HALFTIME_ENGINE]` and `[NCAAB_HALFTIME_PRICED]` for halftime market pricing audit.
- **MLB Prop Engine**: Processes MLB player prop markets (Hits, Total Bases, Batter K, Pitcher K, Hits Allowed, HR, HRR) through a modular pipeline. Key features include a form indicator system, HR qualifying factors (relaxed to 2 minimum from 3), scout-report translation, and a `SignalScore` system for confidence tiers (ELITE, STRONG, SOLID, WATCHLIST, NO_SIGNAL). Signal qualification uses a probability-first approach (≥60% probability gate). The UI uses a signal-first rendering model: all signals (qualified + watch-level) are served to the frontend via `allSignals[]` on the edge cache, ranked and styled by tier rather than hidden. Three UI layers: TopPlays (top 5 by score), LiveBoard (grouped into Elite/Edge/Lean/Watch tiers), and Game Cards (compact single-row layout with signal badges).
- **MLB Intelligence Layer (April 2026)**: Full archetype-driven intelligence pipeline in `server/mlb/`. (1) `archetypes.ts`: 8 batter archetypes (elite_proven, power_first, contact_machine, speed_specialist, back_end, platoon_split, limited_sample, struggling) + 6 pitcher archetypes (ace, mid_rotation, back_end_sp, power_reliever, soft_tosser, unknown_arm) with variance multipliers, PA fragility thresholds, suppression confidence factors, deterioration onset inning, and thesis generation per archetype×market. (2) `marketFamily.ts`: Market family suppression grouping related markets (e.g., hits/total_bases/home_runs/hrr) into families with flagship/derivative ranking. Penalty factors reduce derivative signals (0.92 secondary, 0.85 tertiary). (3) `directionalBias.ts`: Per-market rolling over/under tracking with drift detection and correction factors. Alerts when over/under distribution exceeds 70% in either direction. (4) `calibration.ts`: Archetype×market volatility shrinkage matrix replacing flat 0.92 shrinkage. Safety ceilings enforce hard caps per archetype (no batter market above 85%, HR signals capped at 72%, limited_sample capped at 70%). (5) All wired into `liveGameOrchestrator.ts` pipeline: archetype pre-classification → engine output → safety ceiling → directional tracking → thesis generation → family suppression → cache write. New signal fields: `batterArchetype`, `pitcherArchetype`, `thesis`, `isFlagship`, `familyPenaltyFactor`, `safetyCeilingApplied`, `dataQuality`.
- **MLB Probability Engine (`probabilityEngine.ts`)**: Single canonical probability path (v2.1). Pure probability computation module with ZERO access to book lines, odds, or implied probabilities. Computes model probability from projection + threshold + market variance using market-specific distribution methods: normal CDF (default fallback), negative binomial (TB/HRR with adjustedRate+remainingPA), binomial (batter K/pitcher K/pitcher outs/HR with barrel rate). `calibration.ts` is now a thin shim re-exporting `calibrateProbability`/`applySafetyCeiling` from `probabilityEngine.ts` — retains only `applyDirectionalBias`. Edge calculation is strictly downstream in `markets.ts`: `edge = calibratedProbability - bookImplied`. Purity enforced by `assertProbabilityPurity()` runtime check. All probability in `buildOutput()` and `calculateHitsEdge()` flows through `computeFullModelProbability()`.
- **MLB Phase 2 Engine Rebuild**: Spec-compliant feature layer (`featureEngineering.ts`) with 11 normalized [0,1] scores: contactQuality (includes bat speed at 0.10 weight per spec 2G), batSpeedPower (z-score/tanh conditional amplifier — strong contact 1.0x, neutral 0.55x, weak 0.45x), handednessMatchup, pitchBlendMatchup (damage/whiff modes), hotColdForm (quality-based), parkEnv (market-specific), bvp, lineupOpportunity, bullpenFactor, pitcherSuppression, pitcherDeterioration (5 market variants: hitsAllowed, kDropoff, walksAllowed, hrAllowed, outsRisk). Market engines in `markets.ts` use `applyMarketFeatureWeights()` with per-market formulas (pregameSkill × liveAdj × suppression) for 10 markets: hits (batSpeed 0.04), total_bases (0.16), home_runs (0.20), hrr (0.10), batter_strikeouts (0.06 conditional on whiff+aggression), pitcher_strikeouts, pitcher_outs, hits_allowed, walks_allowed, hr_allowed. Badge system via `computeBadges()` populates canonical `badges[]`/`riskFlags[]` with bat speed tiers: High (score≥0.76), Elite (z≥1.28), Explosive (z≥1.88), Low (z≤-1.0). Integrity firewall (`integrityFirewall.ts`) enforces per-market probability ceilings and qualification floors. Confidence score uses spec 5-component weighted formula: EdgeStrength 0.35, SignalAgreement 0.20, DataFreshness 0.15, LowVarianceProfile 0.15, BadgeSupport 0.15. Event-driven recalculation triggers (inning_change, new_ab, pitcher_change, pitch_count_threshold, tto_shift, lineup_substitution, hard_hit_event) with selective market filtering and reduced dedup window (10s) for high-impact events.
- **MLB UI Architecture (NBA-mirrored rebuild, April 2026)**: Dashboard pill row sub-tabs [Games | Live Feed | HR Radar] controlled from `dashboard.tsx` and passed as `activeSubTab` prop to `MlbLivePage`. **Games tab**: horizontal game chip strip (all today's games as clickable chips showing teams, score, inning, lean badge) + two-panel layout when a game is selected (35% left: game context with score/inning/venue/weather/pitchers; 65% right: active signals via unified `MlbSignalCard`). No 3-level drill-down, no back buttons, no calculator flow. **Live Feed tab**: `TopPlays` (top 6 by signalScore, split over/under) + `LiveBoard` (tier-grouped: Elite/Strong/Solid/Watch, collapsible). **HR Radar tab**: bettable HR edges, HR activity, HR watchlist. **Unified `MlbSignalCard`** (`client/src/components/mlb/MlbSignalCard.tsx`): single component used everywhere — collapsed by default (shows tier/side badges, matchup, player, prop+odds, prob%, edge/proj/score, tags, action buttons), expands on tap to show pitcher arsenal, signal reasons, today's stat line. Signal state machine: ACTIONABLE/ALREADY_HIT/WATCHLIST/STALE with distinct visual treatments. MLB bet slip floats at bottom-right with copy/DraftKings/FanDuel export. UNDER accent is blue (#3b82f6) everywhere. All expandable cards are keyboard-accessible with `role="button"`, `tabIndex`, `aria-expanded`.
- **Engine-First Architecture Patch (April 2026)**: Comprehensive projection integrity layer across NBA and MLB engines. (1) `server/projectionIntegrity.ts`: Shared types (`ProjectionSource`, `ProjectionQuality`, `ProjectionIntegrity`) and assessment functions (`assessMLBProjectionIntegrity`, `assessNBAProjectionIntegrity`). Every surfaced play now carries `projectionSource` (engine_live_context / engine_live_plus_baseline / baseline_only / fallback_static), `projectionQuality` (high/medium/low), `projectionTrustScore` (0-1), `sourceDataCompleteness`, and `fallbackUsed`. (2) NBA engine (`probabilityEngine.ts`): mu bounded by per-stat ceiling per remaining minute (points 1.8/min, rebounds 0.6/min, etc.) preventing projection explosions from pace/usage spikes. `NBAProjectionTrust` with minutesTrustScore and roleTrustScore attached to every engine output. (3) MLB projections (`projections.ts`): Over-regression applied to fragile overs (baseline_weight 0.70, live_weight 0.30) when projection quality is not high. Integrity assessment wired through `buildOutput()` and `calculateHitsEdge()` in `markets.ts`, flowing into `MLBQualifiedSignal` via `liveGameOrchestrator.ts`. (4) Per-game board builder (`topPlaysService.ts`): Replaced global top-10 slice with per-game/per-team board construction. NBA targets 10 plays per team (8-13 range) using A/B/C expansion tiers. MLB builds per-game boards with market quotas (hits 4-8, TB 4-6, pitcher K 1-2, HR 2-4). Duplicate control limits max 3 plays per player. Trust-weighted sorting: tier → trustScore → edge → probability. (5) Confidence from trust: `deriveConfidenceFromTrust()` requires both probability threshold AND projection trust for ELITE/STRONG labels. Fallback-sourced plays cannot reach ELITE. Low-quality projections capped at VALUE. (6) `assignExpansionTier()` gates: Tier A requires trustScore ≥0.7, prob ≥62, no fallback. Tier B requires trustScore ≥0.5, prob ≥58. Tier C requires trustScore ≥0.3, prob ≥55. Below Tier C = REJECT (not surfaced).
- **MLB 50% Probability Fix (April 2026)**: Calculator routes (`/api/mlb/calculate-manual`, `/api/mlb/props`, `/api/mlb/calculate`) spread `...output` from `calculateMLBPropEdge()` which contains `calibratedProbability`, but the frontend reads `probability`/`modelProbability`. Missing field caused fallback to 50% on every calculation. Fixed by adding `probability: output.calibratedProbability` and `modelProbability: output.calibratedProbability` to both route responses.
- **MLB Critical Bug Fixes (March 2026)**: (1) Fixed `sanitizeUserFacingFields()` referencing `MLBLiveGameOrchestrator` instead of `LiveGameOrchestrator` — crashed every signal after qualification. (2) Fixed composite gate in `checkSuppression()` that suppressed ALL signals when contact quality data was null (early season) — now applies 0.5× degraded multiplier to thresholds (TIER1=0.5, TIER3=0.75). (3) Reduced `MLB_MARKET_MIN_GAP` values which were unrealistically high (e.g. batter_strikeouts 0.25→0.05, hits 0.25→0.10) — most player prop projections couldn't exceed the old gap thresholds. (4) Fixed `isPitcherMarket` in `checkSuppression()` missing `pitcher_outs` and `hr_allowed` — those markets were incorrectly applying hitter composite gate. (5) Increased `FEED_FRESHNESS_MS` from 120s to 300s — cache entries were expiring before the API could serve them. (6) **Root cause fix**: edge-feed route depended on `mlbLiveGamesCache` (populated only by `/api/mlb/live-games` endpoint with 30s TTL) to know which game IDs to look up — if that endpoint hadn't been called recently, edge-feed saw 0 games. Fixed by iterating directly over `mlbEdgeCache.entries()` instead. Same fix applied to HR radar route. Together these fixes restored signal flow: 700+ signals per cycle across 9+ live games.
- **MLB Grading (Phase 5)**: End-to-end MLB play grading using MLB Stats API boxscores. `gradePersistedPlays.ts` resolves ESPN event IDs → MLB gamePk via ESPN summary → MLB schedule cross-reference (with team abbreviation normalization: CHW→CWS, ARI→AZ, ATH→OAK). Uses v1.1 feed/live endpoint for game status. MLB_STAT_KEY_MAP maps 14 markets (hits, total_bases, home_runs, hrr composite, batter_strikeouts, pitcher_strikeouts, pitcher_outs, hits_allowed, walks_allowed, earned_runs, outs_recorded, hr_allowed, rbis, walks) to boxscore fields. Grading cron runs every 3 minutes. Admin endpoints: `POST /api/admin/mlb/grade` (manual trigger), `GET /api/admin/mlb/grading-summary` (win rate, ROI, by-market/direction breakdown).
